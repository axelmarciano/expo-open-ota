# Environment variables

The **Expo Open OTA** server requires several environment variables to be set in order to function correctly. These variables are used to configure the server, interact with the Expo API, and manage the server's behavior.
You can set these variables in a `.env` file for local development or in your deployment environment.

:::info Single-app or Multi-app?
**One server can host one or many Expo projects:**

- **Single-app (stateless)** ÔÇö flat env vars (`EXPO_APP_ID`, `EXPO_ACCESS_TOKEN`, `KEYS_STORAGE_TYPE`, ÔÇŽ). Unchanged from v1.
- **Multi-app (control plane)** ÔÇö set `DB_URL` (+ a master key) and manage apps from the dashboard. See [Multi-app](/docs/advanced/multi-app).

The server picks the mode from whether `DB_URL` is set. See the [Expo Apps Configuration](#-expo-apps-configuration) section below.
:::

### ­čô▒ **Expo Apps Configuration**

The server supports one or many Expo apps on the same deployment. Choose the
path that matches how many apps you host ÔÇö both use the exact same runtime.

**Path A ÔÇö single app (keeps v1 env var names unchanged)** ÔÇö set:

| Name | Required | Description | Example |
| --- | --- | --- | --- |
| `EXPO_APP_ID` | Ôťů | The Expo project id (EAS projectId) that clients send as the `expo-app-id` header | `a45ed151-0387-4419-a97c-d008d8f214c7` |
| `EXPO_ACCESS_TOKEN` | Ôťů | Expo access token scoped to that project | `expo_xxx` |
| `KEYS_STORAGE_TYPE` | ÔŁî (defaults to `local`) | `local`, `aws-secrets-manager`, or `environment` | `local` |
| `PUBLIC_LOCAL_EXPO_KEY_PATH` | Ôťů if `KEYS_STORAGE_TYPE=local` | Path to the Expo public key (PEM) | `/keys/public.pem` |
| `PRIVATE_LOCAL_EXPO_KEY_PATH` | Ôťů if `KEYS_STORAGE_TYPE=local` | Path to the Expo private key (PEM) | `/keys/private.pem` |
| `AWSSM_EXPO_PUBLIC_KEY_SECRET_ID` | Ôťů if `KEYS_STORAGE_TYPE=aws-secrets-manager` | AWS Secrets Manager secret ID (public key) | `/eoota/public` |
| `AWSSM_EXPO_PRIVATE_KEY_SECRET_ID` | Ôťů if `KEYS_STORAGE_TYPE=aws-secrets-manager` | AWS Secrets Manager secret ID (private key) | `/eoota/private` |
| `PUBLIC_EXPO_KEY_B64` | Ôťů if `KEYS_STORAGE_TYPE=environment` | Base64-encoded Expo public key (PEM) | `LS0tLS1CRUdJTi...` |
| `PRIVATE_EXPO_KEY_B64` | Ôťů if `KEYS_STORAGE_TYPE=environment` | Base64-encoded Expo private key (PEM) | `LS0tLS1CRUdJTi...` |

**Path B ÔÇö multi-app (control plane)** ÔÇö point the server at a database and it
switches to control-plane mode, where apps live in Postgres and are managed
from the dashboard instead of env vars:

| Name | Required | Description | Example |
| --- | --- | --- | --- |
| `DB_URL` | Ôťů for the control plane | Postgres connection string. Its presence enables control-plane (multi-app) mode; schema migrations run on boot. | `postgresql://user:pass@host:5432/expo_ota` |
| `DB_KEYS_MASTER_KEY_B64` | Ôťů (this **or** the AWS SM var) | 44-char base64 (32-byte) master key that seals per-app signing keys at rest. Generate with `openssl rand -base64 32`. | `Base64 string` |
| `AWSSM_DB_KEYS_MASTER_KEY_SECRET_ID` | Ôťů (this **or** the inline var) | AWS Secrets Manager secret id holding the base64 master key | `/eoota/db-keys-master-key` |

When `DB_URL` is unset the server runs stateless (single-app, Path A). Setting
both master-key variables ÔÇö or neither ÔÇö fails fast at boot. See the
[Multi-app](/docs/advanced/multi-app) guide for the full control-plane setup.

:::caution v2 is a breaking upgrade
The **env var names are identical to v1** on the single-app path, but the
server still requires **client-side changes** (every app must now send an
`expo-app-id` header ÔÇö see the [Migration guide](/docs/advanced/migration-v1-to-v2))
**and** the underlying bucket layout changed from `{prefix}/{branch}/ÔÇŽ` to
`{prefix}/{appId}/{branch}/ÔÇŽ`. Existing v1 bucket data is not read by a v2
server without a one-time data re-path. The Migration guide has the
procedure.
:::

### ­čöÉ **Expo Signing Keys**

Since v2, Expo signing keys are per app: they come from the flat env vars on
the single-app path, or from each app's key configuration (managed in the
dashboard, optionally sealed in the database) on the control plane. CloudFront
keys stay global and are configured via env vars below.
