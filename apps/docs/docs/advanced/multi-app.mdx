# Multi-app (Control Plane)

A single **Expo Open OTA** server can host updates for several Expo projects at
once. Multi-app is served by the **Control Plane** — the server's database-backed
mode. Apps live in Postgres and are created and managed from the dashboard;
clients keep hitting the same server, each request scoped to one app via the
`expo-app-id` header.

:::info When to use the control plane
If you host a single Expo project, stick with the flat env vars documented on the
[Prerequisites page](/docs/getting-started/prerequisites) — that "stateless" mode
is simpler and matches the v1 layout. Switch to the control plane the moment you
need 2+ apps on the same server, or want to add/rotate apps without redeploying.
:::

## The two modes

The server picks its mode at boot from a single signal — whether a database is
configured:

| | Stateless mode | Control Plane (DB mode) |
| --- | --- | --- |
| Trigger | `DB_URL` **unset** | `DB_URL` **set** |
| Apps | Exactly one, from flat env vars (`EXPO_APP_ID`, `EXPO_ACCESS_TOKEN`, key vars) | Many, stored in Postgres |
| Management | Redeploy with new env | Dashboard (add/edit/remove at runtime) |
| Signing keys | File / AWS SM / inline b64 | Same, **plus** `database` (sealed at rest) |

There is no env-var list of apps — the multi-app config lives in the database.

## Enabling the control plane

Set two things and restart:

1. **`DB_URL`** — a Postgres connection string, e.g.
   `postgresql://user:pass@host:5432/expo_ota`. Its presence switches the server
   into control-plane mode; schema migrations run automatically on boot.
2. **A control-plane master key** — exactly one of:
   - `CONTROL_PLANE_MASTER_KEY_B64` — the key inline (local / dev).
   - `AWSSM_CONTROL_PLANE_MASTER_KEY_B64_SECRET_ID` — the AWS Secrets Manager
     secret id holding it (production).

   Setting both, or neither, fails fast at boot.

```bash
DB_URL=postgresql://user:pass@db:5432/expo_ota
CONTROL_PLANE_MASTER_KEY_B64=C1wZdX78be0W+EfSKef/8YhgbPzjeCz0uLMXJskc0h0=
```

### The master key

The master key is a 32-byte symmetric key, provided as a 44-character base64
string. It seals and unseals the per-app signing keys that the `database` key
mode stores in Postgres, so key material is never persisted in plaintext.
Generate one with:

```bash
openssl rand -base64 32
```

:::caution
Losing the master key makes every `database`-mode signing key unrecoverable, and
rotating it re-seals all of them. Store it in your platform's secret store and
back it up independently of the database.
:::

## Managing apps

With the control plane enabled, apps are created and edited from the
**dashboard** (enable it with `USE_DASHBOARD=true`). Creating an app lets you set:

- a display **name** (shown in the app selector; routing still goes by id),
- the app's Expo **access token**,
- the **key storage mode** for its signing keys.

Each app gets its own id (a UUID) — that's the value clients send in the
`expo-app-id` header.

### Key storage modes

| `mode` | Where keys live | When to use |
| --- | --- | --- |
| `local` | Filesystem paths | Keys mounted on the pod / host. |
| `aws-secrets-manager` | AWS Secrets Manager | Fetched on demand with the server's IAM role. |
| `environment` | Base64 PEM you paste in | No file mount available. |
| `database` | Postgres, **sealed** with the master key | Fully self-contained — the default for control-plane apps. |

## Migrating a single-app deploy into the control plane

If you already run a stateless single-app server and want to grow to multi-app,
point it at a database:

1. Provision Postgres and set `DB_URL` + a master key.
2. Keep your existing flat env vars (`EXPO_APP_ID`, `EXPO_ACCESS_TOKEN`, key vars)
   in place for the first boot.
3. On startup the server runs a one-time **infra → DB migration** that seeds your
   existing app (and its branches, channels and update history) into the
   database.
4. From then on, manage that app — and any new ones — from the dashboard. The
   flat env vars are no longer read once the app is in the database.

:::note UUID app id required
The migration requires `EXPO_APP_ID` to be a UUID, because control-plane app ids
are UUIDs. A stateless deploy using a non-UUID id must create the app fresh in
the dashboard instead.
:::

## Client-side header

Every manifest and asset request must carry an `expo-app-id` HTTP header matching
one of your app ids. It's set in the Expo client's `app.config.ts`:

```ts title="app.config.ts"
export default ({ config }) => ({
  ...config,
  updates: {
    url: 'https://ota.mysite.com/manifest',
    requestHeaders: {
      'expo-channel-name': process.env.RELEASE_CHANNEL,
      'expo-app-id': 'a45ed151-0387-4419-a97c-d008d8f214c7',
    },
    // ...
  },
});
```

:::caution Native rebuild required
`updates.requestHeaders` are baked into the **native** bundle at build time
(written into `Expo.plist` on iOS and `AndroidManifest.xml` on Android). Changing
this value — adding, renaming, or removing a header — requires
`npx expo prebuild --clean` + a native rebuild to take effect. Shipping a JS-only
OTA update will **not** propagate a header change. See the
[Migration guide](/docs/migration-v1-to-v2) for the full procedure.
:::

The `eoas` CLI writes this header automatically when you run `npx eoas init`.

## Dashboard

The dashboard shows an **app selector** in the sidebar when more than one app is
configured. Switching apps re-fetches all app-scoped views (branches, channels,
updates) under the newly selected id without a page reload. With a single app the
selector is hidden. If an app has a `name`, the selector displays it — for
example, `Production (a45ed151-…)` — instead of the raw UUID.
