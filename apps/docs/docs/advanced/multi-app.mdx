# Multi-app

A single **Expo Open OTA** server can host updates for several Expo projects at once. You enable this by setting a single structured `EXPO_APPS_JSON` environment variable on the server; clients continue to hit the same server but each request is scoped to one app via the `expo-app-id` header.

:::info When to use multi-app
If you host only one Expo project, stick with the flat env vars documented on the [Prerequisites page](/docs/getting-started/prerequisites) — it's simpler and matches the v1 layout. Switch to `EXPO_APPS_JSON` the moment you need 2+ apps on the same server.
:::

## Resolution order

At boot the server looks for app configuration in this order:

1. **`EXPO_APPS_JSON`** — if set, parse it and register every entry. Flat env vars are ignored in this case.
2. **`EXPO_APP_ID`** — if set (and `EXPO_APPS_JSON` is not), build a single-app config from the flat env vars.
3. **Neither** — the server fails to start with a clear error pointing at both paths.

The two sources are never merged. `EXPO_APPS_JSON` strictly wins, which means adding a second app is a one-env-var edit, not a "delete these old ones and set these new ones" migration.

## JSON shape

Each entry has an `id`, an `accessToken`, and a `keys` block with a `mode` discriminator that selects the other fields. An optional `name` is used as a display label (the dashboard shows it in the app selector instead of the raw `id`).

```json
[
  {
    "id": "a45ed151-0387-4419-a97c-d008d8f214c7",
    "name": "Production",
    "accessToken": "expo_xxx",
    "keys": {
      "mode": "local",
      "publicPath": "/keys/app1-public.pem",
      "privatePath": "/keys/app1-private.pem"
    }
  },
  {
    "id": "b56fe262-1498-5520-ba08-e119e9325d8",
    "name": "Staging",
    "accessToken": "expo_yyy",
    "keys": {
      "mode": "aws-secrets-manager",
      "publicSecretId": "/eoota/app2/public",
      "privateSecretId": "/eoota/app2/private"
    }
  },
  {
    "id": "c67fe373-25a9-6631-cb19-f22afa436e9",
    "accessToken": "expo_zzz",
    "keys": {
      "mode": "environment",
      "publicB64": "LS0tLS1CRUdJTi...",
      "privateB64": "LS0tLS1CRUdJTi..."
    }
  }
]
```

:::note
`name` is optional and display-only. Request routing always goes by `id` — matched against the `expo-app-id` header the client sends — so renaming an app after deployment has no client-side impact.
:::

## Key storage modes

The `keys.mode` field selects how the server resolves an app's Expo signing keys. Setting fields from a different mode than the one declared is rejected at boot.

| `mode` | Required fields | When to use |
| --- | --- | --- |
| `local` | `publicPath`, `privatePath` | Keys on the filesystem — mounted Kubernetes Secret, dev machine, bare metal. |
| `aws-secrets-manager` | `publicSecretId`, `privateSecretId` | Keys live in AWS Secrets Manager; the server fetches them on demand with its IAM role. |
| `environment` | `publicB64`, `privateB64` | Base64-encoded PEM content inlined directly in the JSON — handy when you can't mount a file. |

## Pasting it into your platform

### Railway / Heroku

Paste the JSON into a single `EXPO_APPS_JSON` environment variable. Railway handles multi-line values in the UI, so the JSON can stay formatted; it's stored verbatim. Heroku is the same but via `heroku config:set EXPO_APPS_JSON="$(cat apps.json)"`.

### Render

Paste the JSON into a single `EXPO_APPS_JSON` environment variable. Render's env var UI accepts multi-line values and stores them verbatim.

### Kubernetes / Helm

Define `apps` as a structured list in `values.yaml` and render it into a `Secret`:

```yaml title="helm/templates/apps-secret.yaml"
{{- if .Values.apps }}
apiVersion: v1
kind: Secret
metadata:
  name: {{ include "expo-open-ota.fullname" . }}-apps
type: Opaque
stringData:
  apps.json: |
    {{- toJson .Values.apps | nindent 4 }}
{{- end }}
```

Then mount the Secret and read it into the env var via your entrypoint, or inline it directly in the Deployment:

```yaml
env:
  - name: EXPO_APPS_JSON
    valueFrom:
      secretKeyRef:
        name: {{ include "expo-open-ota.fullname" . }}-apps
        key: apps.json
```

### Fly.io

```bash
fly secrets set EXPO_APPS_JSON="$(cat apps.json)"
```

### Docker / bare metal

Regular env var, `--env-file`, or `docker-compose.yml` `environment:` block.

## Validation rules enforced at boot

The server refuses to start (and prints the offending entry path) if:

- `EXPO_APPS_JSON` is not valid JSON.
- The array is empty.
- An entry's `id` is empty, contains `/`, `\`, or whitespace.
- An entry's `accessToken` is empty.
- `keys.mode` is missing or is anything other than `local`, `aws-secrets-manager`, or `environment`.
- Fields from a different mode than the declared one are set (e.g. `mode=local` with a `publicSecretId`).
- The required fields for the declared mode are missing.
- Two entries share the same `id`.

Errors look like:

```
EXPO_APPS_JSON: apps[2].keys: mode=local requires publicPath and privatePath
```

Always prefixed by the source (`EXPO_APPS_JSON:`) and the path inside the array (`apps[N]`) so you don't have to bisect your config.

## Client-side header

Every manifest and asset request must carry an `expo-app-id` HTTP header matching one of the configured `id`s. This is set in the Expo client's `app.config.ts`:

```ts title="app.config.ts"
export default ({ config }) => ({
  ...config,
  updates: {
    url: 'https://ota.mysite.com/manifest',
    requestHeaders: {
      'expo-channel-name': process.env.RELEASE_CHANNEL,
      'expo-app-id': 'a45ed151-0387-4419-a97c-d008d8f214c7',
    },
    // ...
  },
});
```

:::caution Native rebuild required
`updates.requestHeaders` are baked into the **native** bundle at build time (written into `Expo.plist` on iOS and `AndroidManifest.xml` on Android). Changing this value — adding, renaming, or removing a header — requires `npx expo prebuild --clean` + a native rebuild to take effect. Shipping a JS-only OTA update will **not** propagate a header change. See the [Migration guide](/docs/migration-v1-to-v2) for the full procedure.
:::

The `eoas` CLI writes this header automatically when you run `npx eoas init`.

## Security

:::caution
`EXPO_APPS_JSON` contains access tokens and — in `environment` mode — the private signing keys in plaintext. Treat the env var like any other secret:

- Store it in your platform's secret store (Kubernetes `Secret`, AWS SM, Fly secrets, Railway shared variables).
- Avoid echoing it in CI logs.
- Never commit it to git.

If you need a non-secret file alternative, switch the `keys.mode` to `local` or `aws-secrets-manager` so the JSON only holds pointers, not key material.
:::

## Dashboard

The dashboard shows an **app selector** in the sidebar when more than one app is configured. Switching apps re-fetches all app-scoped views (branches, channels, updates) under the newly selected `id` without a page reload. With a single app the selector is hidden.

If you set the optional `name` field, the selector displays it — for example, `Production (a45ed151-…)` — instead of the raw UUID.
