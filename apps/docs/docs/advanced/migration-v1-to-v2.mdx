# Migrating from v1 to v2

v2 of **Expo Open OTA** introduces multi-app support on a single server. On the single-app path the server-side env var names are unchanged, and **your already-installed clients keep receiving updates without a rebuild**.

Two things change:

1. The **bucket/storage layout changed** from `{prefix}/{branch}/{runtimeVersion}/{updateId}/…` to `{prefix}/{appId}/{branch}/{runtimeVersion}/{updateId}/…`. The v2 server re-paths your data automatically on first boot — see [Automatic bucket re-path](#single-app-deployments) below.
2. v2 identifies an app by the **`expo-app-id` HTTP header** instead of by the deploy's env var. Clients built against v1 do not send it, so on a single-app deploy the server falls back to `EXPO_APP_ID` — the only app such a deploy has.

:::tip Your v1 clients keep working — no rebuild needed
`requestHeaders` are baked into `Expo.plist` / `AndroidManifest.xml` at build time, so an already-installed v1 binary can never start sending `expo-app-id` without a store release. Requiring it would mean an OTA server that needs a store round-trip to keep delivering OTA updates.

So when the header is absent, v2 resolves the request to `EXPO_APP_ID`. There is nothing to disambiguate: a single-app deploy has exactly one app. Adding the header to `app.config.ts` and rebuilding is **optional**, and only becomes necessary the day you host a second app on the same server.
:::

:::warning The bucket re-path is not zero-downtime
Client-side you are fine, but the storage move is not free: on S3/GCS you must stop publishes while it runs, and there is no opt-out. See [Take writers offline](#single-app-deployments) below.
:::

## What changed on the wire

| Concern | v1 | v2 |
| --- | --- | --- |
| Server identifies the app by | `EXPO_APP_ID` env var (one per deploy) | `expo-app-id` HTTP header sent by the client, falling back to `EXPO_APP_ID` when the header is absent (v1 clients) |
| Config shape | Flat env vars | Flat env vars (single-app, unchanged) **OR** the control plane (`DB_URL`, multi-app) |
| Manifest/asset URL | `/manifest`, `/assets`, `/rollback/{branch}`, … | Same for manifest/assets (`expo-app-id` comes in a header). Publish/rollback/republish/upload routes are now scoped under `/{APP_ID}/…` — handled transparently by the `eoas` CLI. |
| Dashboard API | `/api/branches`, `/api/channels`, `/api/branch/{branch}/…` | `/api/apps/{APP_ID}/branches`, `/api/apps/{APP_ID}/channels`, `/api/apps/{APP_ID}/branch/{branch}/…` |

## Step-by-step migration

### Single-app deployments

1. **Upgrade the server image** to the v2 release. No env var changes are required on the single-app path — v2 reads the same `EXPO_APP_ID`, `EXPO_ACCESS_TOKEN`, `KEYS_STORAGE_TYPE`, and key-storage env vars as v1.

    :::tip Automatic bucket re-path
    On first boot, the v2 server runs the `20260422_v2_scope_data_under_appid` migration which **automatically moves your v1 bucket data into the new `{appId}/…` layout**. It only touches directories that match the v1 branch shape (`{branch}/{runtimeVersion}/{updateId}/.check`), leaves `.migrationhistory` at the bucket root, and records its success in the migration ledger so it runs exactly once. An interrupted run picks up where it left off on the next boot.

    There is **no opt-out**: upgrading to v2 means accepting the re-path. If the migration fails, the server does not boot — fix the cause (usually bucket credentials or permissions) and boot again. The move is idempotent, so a run interrupted anywhere converges on retry.

    The migration is **only** run when `EXPO_APP_ID` is set (single-app stateless path). Control-plane (DB mode) deploys don't set `EXPO_APP_ID`, so the re-path no-ops there — a fresh v2 control-plane install has no v1 root-level data to move.
    :::

    :::warning Take writers offline during the automatic S3/GCS re-path
    On object-storage backends the migration uses a two-phase listing: it first scans the bucket to confirm every v1 `(branch, runtimeVersion, updateId)` triple, then copies + deletes each object into the `{appId}/` layout. **Any publish / republish / rollback that lands between those two phases — or after phase 2 for a new triple — will not be picked up**, and its files stay at the old root. Stop CI publishes and point clients away from the bucket until the migration completes and the log prints `✅ v1→v2 bucket re-path complete.` The local filesystem backend is not affected (it reads the root once and `os.Rename` per entry).
    :::

    :::note App-id / v1-branch name collision
    If your target `EXPO_APP_ID` matches the name of an existing v1 branch, the migration refuses to run and boot fails with `app id collides with a v1 branch of the same name` — moving the branch under an id equal to its own name would nest the data under itself. Rename that branch in the bucket, then boot again to retry.

    `EXPO_APP_ID` is an Expo project id and therefore a UUID, so in practice this only fires if you were running v1 with a hand-written id.
    :::

2. **Upgrade the `eoas` CLI** (`npm i -D eoas@latest`). This one is required: publish, rollback and republish now go through `/{APP_ID}/…` routes, and a v1 CLI gets a `404` against a v2 server. Unlike the client binary, the CLI upgrades with an `npm install` — no store round-trip, no user tail.

3. **Check that your v1 clients are covered**, before you go any further:

    ```bash title="cd ./my-expo-project"
    npx eoas doctor --channel=production
    ```

    It sends two manifest requests to your server — one shaped like a v1 client (no `expo-app-id`) and one like a v2 client — and reports whether each is served. If the v1 probe passes, your installed fleet keeps updating and **you are done migrating**. Steps 4 to 6 are optional.

4. **(Optional) Add `expo-app-id` to every client app's `app.config.ts`:**

    ```ts title="app.config.ts"
    export default ({ config }) => ({
      ...config,
      updates: {
        url: 'https://ota.mysite.com/manifest',
        requestHeaders: {
          'expo-channel-name': process.env.RELEASE_CHANNEL,
          // New in v2 — value must match the server-side EXPO_APP_ID
          'expo-app-id': 'a45ed151-0387-4419-a97c-d008d8f214c7',
        },
        // ...
      },
    });
    ```

5. **(Optional) Regenerate native code and rebuild each platform:**

    ```bash title="cd ./my-expo-project"
    npx expo prebuild --clean        # rewrites ios/ and android/ from app.config.ts
    npx expo run:ios                 # or run:android — rebuild the native binary
    ```

    After this, verify that `ios/<app>/Supporting/Expo.plist` contains the new header:

    ```xml
    <key>EXUpdatesRequestHeaders</key>
    <dict>
      <key>expo-channel-name</key>
      <string>production</string>
      <key>expo-app-id</key>
      <string>a45ed151-0387-4419-a97c-d008d8f214c7</string>
    </dict>
    ```

    Same check on Android: `android/app/src/main/AndroidManifest.xml` must carry a matching `meta-data` entry for `expo.modules.updates.EXPO_UPDATES_REQUEST_HEADERS`.

6. **(Optional) Ship the new native build** through the App Store / Play Store / internal distribution, on your normal release cadence. There is no rush and no deadline: clients on the v1 binary keep being served through the `EXPO_APP_ID` fallback for as long as you leave it on.

Regular OTA updates resume as usual via `npx eoas publish --branch=<branch>` as soon as step 1 completes.

:::tip Tightening once the fleet has turned over
Once `expo-app-id` ships in every build your users run, you can require it: set `SKIP_LEGACY_APP_ID_FALLBACK=true` and header-less requests go back to `400 No app id provided`. This is worth doing before hosting a second app on the deploy, and it is how you surface stragglers still running a v1 binary.

Check the boot log to see which state you are in — the server prints the app the fallback resolves to, or that it is disabled.
:::

### Migrating to multi-app (control plane)

Once you're on v2 and want to host a second Expo project on the same server, move
to the **control plane** — multi-app lives in a database, not env vars:

1. Provision Postgres and set `DB_URL` plus a master key for the database-stored
   signing keys (`DB_KEYS_MASTER_KEY_B64` or `AWSSM_DB_KEYS_MASTER_KEY_SECRET_ID`).
   Keep your existing flat env vars in place for the first boot.
2. On startup, a one-time infra → DB migration seeds your existing app (and its
   branches, channels and update history) into the database. From then on, manage
   apps from the dashboard.

    :::info Your signing keys move into the database
    If your app uses `KEYS_STORAGE_TYPE=local` or `environment`, the migration
    reads your **existing** key pair and seals it into the database under the
    master key, switching the app to `database` key mode. The key pair itself is
    preserved byte for byte — your already-installed builds keep verifying signed
    manifests, and no rebuild is needed.

    This is why the master key must be set *before* the first boot. If it is missing
    the server refuses to start at all, before the migration runs; if it is present
    but malformed, the migration aborts and rolls back rather than storing an app
    with no usable key. Either way nothing is recorded as applied — fix the key and
    boot again.

    Once migrated, you no longer need `PUBLIC_LOCAL_EXPO_KEY_PATH` /
    `PRIVATE_LOCAL_EXPO_KEY_PATH` (or the `_B64` vars), and you no longer need to
    mount the PEM files on every replica. Keep a backup of the original keys anyway:
    the migration does not delete them, and there is no down migration.

    `KEYS_STORAGE_TYPE=aws-secrets-manager` is left untouched — the app keeps
    fetching its keys from AWS, since a secret id stays reachable from every
    replica.
    :::
3. Add the second app from the dashboard (name, access token, key storage), then in
   its `app.config.ts` set `'expo-app-id'` to the new app's id and run
   `npx eoas init` to wire the rest (URL, certificate, channel header). Rebuild
   natively as described above.
4. Publish updates targeting the new app: the `eoas` CLI picks up the right
   `expo-app-id` from the config automatically.

See the [Multi-app](/docs/advanced/multi-app) guide for the full control-plane
reference (including the UUID app-id requirement for the automatic migration).

## Manual bucket re-path

The automatic migration handles this for you and cannot be turned off. These commands are for **recovery** — a re-path that died partway through and left objects at the old root, or data the structural detection did not classify as v1. Run them against a stopped server, then boot: the migration re-runs, finds nothing left at the root, and records itself as applied.

Replace `<APP_ID>` with the exact value of your `EXPO_APP_ID`, and **take the server offline before starting** to prevent concurrent writes into the old paths.

```bash title="Local filesystem"
cd "$LOCAL_BUCKET_BASE_PATH"
# Only move the branch directories, not any existing .migrationhistory file.
mkdir -p "<APP_ID>"
for d in */; do
  [ "$d" = "<APP_ID>/" ] && continue
  mv -- "$d" "<APP_ID>/"
done
```

```bash title="S3 (AWS CLI)"
# Dry-run first to preview the moves, then drop --dryrun.
aws s3 mv "s3://$S3_BUCKET_NAME/" "s3://$S3_BUCKET_NAME/<APP_ID>/" \
  --recursive --exclude ".migrationhistory" --dryrun
```

```bash title="GCS (gsutil)"
gsutil -m mv "gs://$GCS_BUCKET_NAME/**" "gs://$GCS_BUCKET_NAME/<APP_ID>/"
```

After the move, your layout should be `{prefix}<APP_ID>/{branch}/{runtimeVersion}/{updateId}/…`. Leave `.migrationhistory` at the bucket root — the server reads it from there.

## Troubleshooting

### Client gets `400 No app id provided`

The client sent no `expo-app-id` **and** the server had no legacy app to fall back to. Run `npx eoas doctor` to see which of the two it is, then:

- **On a single-app deploy**: either `SKIP_LEGACY_APP_ID_FALLBACK` is set (turn it off, or rebuild the client with the header), or `EXPO_APP_ID` is unset — but then the server would not have booted at all, so check your env.
- **On the control plane** (`DB_URL` set): there is no fallback by design, since a control-plane deploy has several apps and a header-less request cannot be attributed to one. The client must send the header. Check `ios/<app>/Supporting/Expo.plist` and `android/app/src/main/AndroidManifest.xml` — if the header isn't there, you set `requestHeaders` in `app.config.ts` but haven't run `npx expo prebuild --clean` and rebuilt the native binary.
- **On a control plane migrated from a v1 deploy**: keep `EXPO_APP_ID` set in the env. It names the app your legacy clients belong to, and the infra → DB migration preserves it as the app's id.

### Client gets `400 Error validating expo auth`

- The `expo-app-id` header value doesn't match any configured app on the server. Double-check that:
  - On single-app (stateless): `EXPO_APP_ID` on the server equals the `expo-app-id` in the client's `app.config.ts`.
  - On the control plane: an app with that id exists in the dashboard and matches the client's `expo-app-id`.

### Server fails to boot with `no app config found`

- `EXPO_APP_ID` is not set (stateless single-app), and no `DB_URL` is configured (control plane). Set one path per the [Prerequisites page](/docs/getting-started/prerequisites) (single-app) or [Multi-app](/docs/advanced/multi-app) page.

### Dashboard shows `404 page not found`

- The dashboard is hitting the new `/api/apps/{APP_ID}/…` routes. If you see 404s after a v2 upgrade, you're likely running the v1 dashboard bundle against a v2 server — redeploy the server (which ships the updated dashboard bundle) or clear your browser's cached asset for `/dashboard/`.
